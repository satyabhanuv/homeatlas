# v2.7.9 QA Runbook — Level 4 Pre-Geocoded KV + Haversine Radius Filter

_Post-deploy verification steps. Run in order. Every step has a pass/fail check. Do not proceed to the next step if the current one fails — send me the error output and I'll fix._

---

## Pre-flight: what needs to be deployed

Deploy in this order:

**1. Worker (`nearnity-events-worker.js`) — critical**
- Cloudflare dashboard → Workers & Pages → `nearnity-events` → Edit code → paste `nearnity-events-worker.js` → Save & Deploy
- OR: `npx wrangler deploy` if you have local wrangler set up

**2. `ADMIN_TOKEN` secret (required to run ingest)**
- Cloudflare dashboard → Workers & Pages → `nearnity-events` → Settings → Variables → Add variable
- Name: `ADMIN_TOKEN`
- Value: a random string. Generate one: `openssl rand -hex 32` in Terminal, or use any password generator. Save this string — you'll paste it into URLs below.
- Check "Encrypt" so it's stored as a secret.

**3. Frontend (`index.html`) — after Worker is confirmed working**
- Push to GitHub → Cloudflare Pages auto-deploys ~1 min
- Hard-refresh nearnity.com. Footer should read **v2.7.9**

## QA Step 2: Verify ADMIN_TOKEN gate works

Open: `https://nearnity.com/api/admin/geo-ingest?state=CA` (no token)

**Expected:** `{"error":"Forbidden"}` with status 403.

Then: `https://nearnity.com/api/admin/geo-ingest?state=CA&admin_token=WRONG_VALUE`

**Pass:** both requests return 403.
**Fail:** if either returns 200 or an ingest report → your ADMIN_TOKEN isn't set on the Worker. Redo Pre-flight step 2.

## QA Step 3: Run California ingest (takes ~30-90 seconds)

Replace `YOUR_TOKEN` with the string you saved:

`https://nearnity.com/api/admin/geo-ingest?state=CA&admin_token=YOUR_TOKEN`

Wait for the response. It'll take a while (fetching CMS + NPPES + running Census Batch Geocoder).

**Expected (roughly):**
```json
{
  "state": "CA",
  "started_at": "2026-07-08T...",
  "cms_fetched": 358,
  "cms_geocoded": 352,      ← at least 90% match rate
  "cms_written": 352,
  "nppes_fetched": 5000,
  "nppes_geocoded": 4850,
  "nppes_bucketed": {
    "ER": 45, "UC": 380, "CLINIC": 2100, "DENTIST": 890, "PHARMACY": 570
  },
  "elapsed_ms": 45000,
  "ok": true
}
```

**Pass:**
- `ok: true`
- `cms_written >= 300`
- `nppes_bucketed.ER >= 20`
- `nppes_bucketed.UC >= 200`
- `elapsed_ms < 120000` (2 min)

**Fail cases:**
- If `cms_geocoded / cms_fetched < 0.7` → Census Batch geocoder had a bad day. Retry in 5 min.
- If `ok: false` with error → paste the error to me.
- If it times out at 30 seconds (Cloudflare Worker limit) → we need to chunk the work. I'll ship v2.7.9.1 with a bounded per-request cap and a "run again to resume" pattern.

## QA Step 5: Test the radius endpoint directly (5 fixtures)

### Fixture A — San Jose (Corbal Ct)
`https://nearnity.com/api/medical-radius?lat=37.3175&lon=-121.7891&state=CA&radius_mi=10&taxonomy=all&limit=30`

**Expected:**
- `meta.kv_hit: true`
- `count >= 15`
- `results[0].distance_mi` should be under 5 mi
- Results include at least one hospital (`category: "er"` or `"hospital"`) and multiple urgent cares
- Regional Medical Center of San Jose likely appears (well-known local ER)

### Fixture B — San Jose (Ellicott Loop)
`https://nearnity.com/api/medical-radius?lat=37.4419&lon=-121.9165&state=CA&radius_mi=15&taxonomy=er`

**Expected:**
- `kv_hit: true`
- `count >= 3` (at least a few ERs in that radius)
- All results have `category: "er"` or `emergency_services: true`
- `distance_mi` sorted ascending

### Fixture C — Urgent care only
`https://nearnity.com/api/medical-radius?lat=37.3175&lon=-121.7891&state=CA&radius_mi=10&taxonomy=urgent`

**Expected:**
- `kv_hit: true`
- `count >= 8` (Bay Area has dense UC coverage)
- All results `category: "urgent"`
- Names likely include Concentra, One Medical, Kaiser Urgent, Sutter Walk-In etc.

### Fixture D — State without ingest (should trigger fallback signal)
`https://nearnity.com/api/medical-radius?lat=44.0682&lon=-114.7420&state=ID&radius_mi=20`

**Expected:**
- `meta.kv_hit: false`
- `meta.fallback_recommended: true`
- `count: 0`
- `note` includes "Run /api/admin/geo-ingest?state=ID to populate"

### Fixture E — Latency benchmark
Repeat fixture A five times. Note `meta.elapsed_ms` each time.

**Expected:** all under 200ms, most under 100ms (first hit warms Cloudflare's edge cache; subsequent hits should be ~30-50ms).

## QA Step 6: End-to-end frontend test

Open **nearnity.com** in a fresh browser (or Incognito). Footer must read **v2.7.9**.

DevTools open (Preserve log ON). Search "ER" or "urgent care" near 3191 Corbal Ct San Jose (or your address).

**Expected Console output:**
```
[federal-medical] v2.7.9 KV path hit: 30 results in 45ms from states=[CA]
[public-services] OSM returned X elements, federal returned 30 items
```

**Expected Network tab:**
- ONE call to `/api/medical-radius?lat=...&state=CA&...` — status 200
- No `/api/medicare-quality` or `/api/nppes` calls (those only fire on KV miss)
- Response body has `meta.kv_hit: true`

**Expected UI:**
- ER tab: shows anchor card + real hospital results (Regional Medical Center of San Jose etc.) with distance labels
- Urgent care tab: shows anchor card + Bay Area urgent care chains (Concentra, One Medical, etc.)
- Hospitals tab: shows non-ER hospitals + rating info
- All tabs populate. No skeletons stuck.

**Pass:** Console shows KV path hit + tabs populate.
**Fail modes:**
- Console shows "KV miss for state CA" → the ingest metadata wasn't written correctly. Re-run step 3.
- Console shows "medical-radius threw" → send me the error message.
- Tabs still empty even though Console shows "KV path hit: X results" → merge logic broke. Send me the Console log + a screenshot.

## Rollback if v2.7.9 breaks something worse

If v2.7.9 introduces a regression worse than v2.7.8.1, quick rollback:

**Frontend rollback:** git revert the v2.7.9 index.html commit → Cloudflare Pages auto-redeploys prior version.

**Worker rollback:** Cloudflare dashboard → Workers → nearnity-events → Deployments tab → find previous deploy → click "Rollback".

The frontend has a graceful fallback baked in: if `/api/medical-radius` returns error/miss, it uses the old Medicare + NPPES calls. So worst-case behavior is v2.7.8.1 equivalent, not total breakage.
